# AI Account Switcher v0.5.10

This release makes Claude and Codex session priming reliable from the macOS GUI and background
worker, while keeping the UI honest about what the provider has actually confirmed.

## What changed

- Fixed Codex manual and scheduled prime failures caused by missing GUI/LaunchDaemon `PATH`
  entries (`CLI exit 127`).
- Fixed Codex rejecting background primes outside a Git repository. Prime runs are now ephemeral,
  read-only, use the account profile as their working directory, and skip the repository check.
- Restored **Prime ngay** for Claude accounts whose five-hour window is not active. Unknown,
  authentication-error, and unreadable quota states still fail closed.
- Changed pending UI text to **request sent, awaiting confirmation**. The app only says a session
  opened after it verifies the provider's reset state.
- Added isolated Claude recovery for quota HTTP 401/429: run one safe CLI refresh, reload the token,
  and retry once, with a five-minute per-account cooldown.
- Prevented background Claude prime/token-refresh commands from loading project, plugin, hook, MCP,
  history, or shared config paths that can trigger macOS permission prompts for protected folders.
- Hardened Claude further by disabling the built-in tool/sandbox layer and all context sources;
  safe mode by itself still allowed macOS protected-folder preflights.
- Changed Codex priming to call the Codex response endpoint directly with the selected profile's
  OAuth token. It no longer starts `codex exec`, Git discovery, or an agent sandbox just to send
  the minimal prime request.
- Added concise CLI stderr details to prime failures instead of reporting only an exit code.

## Sleep, lock, and multiple macOS users

The optional headless prime worker is installed per macOS user and uses that user's app state,
`HOME`, and account profile directories. It does not use another logged-in user's accounts.

The app never stores or types the macOS login password. If the Mac sleeps and the user's login
Keychain remains locked, credential access may fail; the attempt is released and can retry after
that user unlocks the Mac. This is expected macOS security behavior.

## Prime result semantics

- **Pending:** the request was sent; confirmation is still in progress.
- **Success:** the provider reset state proves an active/fixed five-hour window.
- **Hold:** an existing window is still active.
- **Failed/unknown:** the request failed, quota could not be read, or a new window could not be
  proven. The app does not claim a session opened in these cases.

## Install

Download `AI.Account.Switcher_0.5.10_universal.dmg` from the release assets and drag the app into
Applications.

The build is not code-signed. On first launch, right-click the app and choose **Open**, or run:

```bash
xattr -cr "/Applications/AI Account Switcher.app"
```
